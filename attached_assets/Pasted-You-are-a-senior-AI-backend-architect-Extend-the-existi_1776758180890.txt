You are a senior AI/backend architect. Extend the existing Fastify-based Node.js system to implement **LLM-enhanced API ↔ DB lineage mapping using RAG (Retrieval-Augmented Generation)**.

This MUST enhance the existing deterministic lineage system using ChromaDB + LLM reasoning and include robust validation, caching, and observability.

---

# 🎯 OBJECTIVE

Improve lineage accuracy by:

1. Retrieving relevant code + schema from Chroma
2. Sending structured context to LLM
3. Inferring API → Function → DB table relationships
4. Merging results with deterministic lineage
5. Ensuring reliability, repeatability, and traceability

---

# ⚙️ LLM SETUP (MANDATORY)

Use OpenAI (or compatible API)

### llm.service.ts

Function:

```ts
generate(prompt: string): Promise<string>
```

### REQUIRED FEATURES

* Retry mechanism (max 3 retries)
* Timeout handling
* Structured logging for each request
* Response size guard

# 🧠 RAG FLOW

## Step 1: Input

* API endpoint (route + method)
* Existing deterministic lineage

---

## Step 2: Retrieve Context (Chroma)

Query using:

* API route
* function names
* domain keywords

### REQUIREMENTS

* Limit results (top 5–10 chunks)
* Deduplicate chunks
* Log retrieved sources

---

## Step 3: Build Prompt (MANDATORY STRUCTURE)

Use prompt.service

Template:

```text
You are a senior software architect.

Given the following API and system context, identify:

1. Database tables used
2. Operations (SELECT, INSERT, UPDATE, DELETE)
3. Execution flow (function chain)

API:
{api}

Code Context:
{chunks}

Database Schema:
{schema}

Return STRICT JSON:
{
  "tables": [
    { "name": "string", "operation": "string" }
  ],
  "flow": ["string"]
}
```

---

## Step 4: LLM CALL (WITH RETRY)

* Retry up to 3 times if:

  * invalid JSON
  * empty response
* Log each attempt

---

## Step 5: VALIDATE OUTPUT (MANDATORY)

* Strict JSON parsing
* Reject malformed output
* Normalize:

  * table names (lowercase)
  * operations (uppercase)

Fallback:

* If LLM fails → return deterministic result

---

## Step 6: MERGE RESULTS

Combine:

* deterministic lineage
* LLM inference

### RULES

* Match → confidence = HIGH
* LLM-only → confidence = LOW
* Conflict → mark as CONFLICT

---

## Step 7: CACHE RESULTS (MANDATORY)

Use:

/services/cache.service.ts

Cache key:

```ts
projectId + apiId + promptVersion
```

Behavior:

* Return cached result if exists
* Store LLM output after success

---

## Step 8: STORE RESULTS

Extend table:

api_table_map

Add fields:

* source (deterministic | llm | merged)
* confidence (high | medium | low | conflict)
* prompt_version
